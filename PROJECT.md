# 财务助手项目（AccoBot）

## 项目目标

构建一个智能财务助手系统，帮助用户高效完成日常财务工作，降低财务管理门槛，提升工作效率和准确性。

## 核心功能规划

### 1. 原始凭证管理
- 凭证录入（手动/OCR识别）
- 凭证分类与归档
- 凭证查询与检索
- 附件上传与关联

### 2. 做账
- 会计分录自动生成建议
- 科目智能匹配
- 凭证审核与过账
- 期末结转

### 3. 对账
- 银行对账单导入
- 自动对账匹配
- 差异标记与处理
- 对账报告生成

### 4. 报税
- 税种管理（增值税、企业所得税、个税等）
- 税额自动计算
- 申报表生成
- 报税日历提醒

### 5. 账簿查看
- 总账
- 明细账
- 日记账
- 科目余额表

### 6. 报表
- 资产负债表
- 利润表
- 现金流量表
- 自定义报表

### 7. 数据分析
- 收支趋势分析
- 费用结构分析
- 利润分析
- 异常数据预警
- 可视化图表

## 技术选型

| 层面 | 选型 | 说明 |
|------|------|------|
| 语言 | Python 3.11+ | 与 Hermes Agent 一致，生态丰富 |
| 包管理 | uv + pyproject.toml | 快速、现代的 Python 包管理 |
| LLM 调用 | openai SDK | 兼容 OpenAI/国产模型 API |
| 数据库 | SQLite（WAL 模式） | 零安装、单文件、事务安全；核心字段用列存储保证聚合查询性能，差异化字段用 JSON 列保持灵活性 |
| Web 后端 | FastAPI + uvicorn | 异步高性能，原生支持 WebSocket |
| Web 前端 | Vue 3 或原生 HTML（待定） | Windows 原生支持，不依赖 PTY |
| 通信 | REST API + WebSocket | REST 做数据 CRUD，WebSocket 做对话流式输出 |
| 测试 | pytest | 标准选择 |

### 数据库设计思路

采用 SQLite + JSON 字段的混合方案：
- **核心字段**（金额、科目代码、日期）用列存储 → 聚合查询快、事务安全、支持索引
- **差异化字段**（不同行业/用户的特殊字段）用 JSON 列 → 灵活度高，SQLite 3.38+ 原生支持 `json_extract()` 查询
- **后续扩展**：如需多用户 Web 服务，可平滑迁移到 PostgreSQL（JSONB 类型，SQL 语法高度兼容）

数据库文件规划：
- `master.db` — 全局元信息：用户角色、账套列表、AI 偏好配置
- `chat.db` — 聊天会话、消息历史、Agent 记忆（参考 Hermes 的 `hermes_state.py`）
- `accounting_{company_id}.db` — 每家企业独立的账务数据（凭证、分录、科目、账簿），一家一个文件，隔离彻底
- `config.yaml` + `.env` — 系统配置和 API 密钥

### Hermes Agent 架构总结

**定位：** 自我进化的通用 AI Agent，支持多模型、多平台、多工具，MIT 协议。

**核心架构：**

```
run_agent.py          → AIAgent 类，核心对话循环（同步，OpenAI 消息格式）
model_tools.py        → 工具编排，发现/注册/调度所有工具
toolsets.py           → 工具集定义，按场景分组工具
tools/registry.py     → 工具注册中心（所有工具文件自动发现注册）
tools/*.py            → 具体工具实现（40+内置工具）
agent/                → Agent 内部模块（提供商适配、记忆、压缩、提示词构建等）
hermes_cli/           → CLI 子命令、配置、插件加载
gateway/platforms/    → 消息平台适配器（Telegram/Discord/Slack/微信等）
plugins/              → 插件系统（记忆/模型/看板/图片生成等）
skills/               → 技能系统（可学习、可创建、可自我改进的程序化记忆）
cron/                 → 定时任务调度
```

**关键设计模式（可借鉴）：**

1. **工具注册机制** — `tools/registry.py` 统一注册，工具文件自动发现，handler 返回 JSON 字符串
2. **工具集（Toolset）** — 按场景组合工具，支持继承和组合
3. **对话循环** — 同步循环 + 迭代预算 + 中断检查，工具调用结果回注消息列表
4. **插件系统** — 生命周期钩子（pre/post tool call, session start/end），用户可扩展
5. **技能系统** — Markdown 格式的程序化知识，可被 Agent 自动创建和改进
6. **记忆管理** — 可插拔的记忆后端（ABC 接口），支持多种存储
7. **多平台网关** — 统一消息接口，每个平台一个适配器
8. **配置分层** — config.yaml（设置）+ .env（密钥），支持 Profile 多实例隔离
9. **子代理委派** — `delegate_task` 生成隔离子代理并行处理

**技术栈：**
- Python 3.11+，openai SDK，httpx，rich，prompt_toolkit，pydantic
- SQLite（会话存储 + FTS5 搜索）
- 支持 OpenAI/Anthropic/各种国产模型
- 包管理用 uv，测试用 pytest

## 新想法记录

### 用户角色与多账套架构

Bot 首次使用时必须明确用户角色，因为不同角色的服务侧重点和数据架构完全不同：

| 角色 | 特征 | 服务侧重 | 数据架构 |
|------|------|---------|---------|
| 小企业负责人 | 单账套，不懂会计 | 教学引导、极简操作、风险提示 | 单账套，简单结构 |
| 大企业专业会计 | 单账套，专业用户 | 效率工具、批量操作、深度分析 | 单账套，完整功能 |
| 代账公司员工 | 多账套，专业用户 | 多账套快速切换、批量报税、统一管理 | 多账套隔离，每家企业独立 |

**数据隔离方案：** 采用"一家企业一个独立数据库文件"的方式：
- 每个账套对应一个独立的 `accounting_{company_id}.db` 文件
- 优势：隔离彻底、备份简单（复制文件）、删除干净、不会串数据
- 代账模式下有一个 `master.db` 管理所有账套元信息（公司名、创建时间、状态等）
- 切换账套 = 切换数据库连接，零风险

**角色影响 AI 行为：**
- 小企业负责人：默认用大白话，主动教学，操作前多确认
- 专业会计：默认用术语，减少解释，支持快捷操作
- 代账员工：强调效率，支持批量操作，跨账套汇总视图

### 学习辅导功能

Bot 不仅是工具，还是老师。用户在处理财务工作中遇到疑问或不明白的地方时：
- 自动记录问题到"学习区域"
- 关联对应的知识点（会计准则条文、税法规定、操作规范等）
- 跟踪用户对每个知识点的掌握情况（未学习/已了解/已掌握）
- 下次遇到类似问题时，根据掌握情况决定是直接处理还是再次讲解
- 用户可主动查看学习记录，复习薄弱知识点

### 质检与风控

Bot 对用户的每一笔操作进行风险审查，主动提出问题：
- **企业风控** — 超预算支出、异常大额交易、关联交易、费用报销超标等
- **国家政策** — 不符合会计准则的处理方式、科目使用错误、核算方法不当
- **法律合规** — 涉税风险（虚开发票、进项抵扣不合规）、反洗钱规则、资金往来合规性
- **数据一致性** — 借贷不平、跨期入账、重复录入等技术性错误

质检结果分级：
- 🔴 严重（必须处理，否则可能违法违规）
- 🟡 警告（建议处理，存在风险）
- 🟢 提示（最佳实践建议，可忽略）

### 产品设计原则

**用户是小白，不是程序员。** 所有交互必须遵循：
- 不要求用户做任何代码操作或手动编辑配置文件
- 所有设置通过 Web UI 引导完成（API Key、账套创建、科目选择、Gateway 平台配置等）
- 安装和启动尽量一键化，减少步骤
- 错误提示用人话，不暴露技术细节
- 默认值要合理，让用户尽量少做选择
- Playwright 作为默认依赖安装（不是可选），因为与政府/银行网站交互是财务刚需
- 消息平台（企业微信/钉钉/飞书）的配置通过 Web UI 完成，体验与 API Key 配置一致
