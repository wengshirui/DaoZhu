# 051 — 多角色 Agent（调度/执行/质检）

**Intake ID:** 2026-05-28-multi-role-agent
**Requester:** PO
**Urgency:** Normal
**Status:** 待澄清

---

## 原始需求

> 大模型在执行用户对话的时候，能不能分为三个角色：调度、执行、质检。最少循环一次，保证执行质量。

## 提取意图

当前 Agent 是单一角色（一个 LLM 调用循环），容易出现：
- 不规划就乱调工具
- 执行完不检查结果质量
- 工具调用次数浪费

改进为三角色协作模式：
1. **调度（Planner）** — 分析用户意图，制定执行计划（调哪些工具、什么顺序）
2. **执行（Executor）** — 按计划逐步执行工具调用
3. **质检（Reviewer）** — 检查执行结果是否满足用户需求，不满足则反馈给调度重新规划

最少循环一次（调度→执行→质检），保证输出质量。

## 开放问题

| # | 模块 | 问题 | AI 猜测 | 答案 |
|---|------|------|---------|------|
| 1 | 成本 | 三角色意味着至少 3 次 LLM 调用（规划+执行+质检），对 token 消耗和响应速度的影响如何权衡？ | 简单问题（搜索、查询）跳过质检；复杂任务（创建工作区、多步操作）才走完整流程 | |
| 2 | 触发条件 | 什么情况下走三角色流程，什么情况下走简单单次回复？ | 用户意图涉及工具调用时走三角色；纯聊天/问答直接回复 | |
| 3 | 质检标准 | 质检角色用什么标准判断"执行质量合格"？ | 检查：1)是否完成了用户请求 2)是否有错误 3)结果是否合理 | |
| 4 | 循环上限 | 调度→执行→质检不通过→重新调度，最多循环几次？ | 最多 2 次循环（调度→执行→质检→重调度→重执行→终审），防止无限循环 | |
| 5 | 用户感知 | 三个角色的工作过程是否对用户可见？还是只展示最终结果？ | 可见：调度阶段显示"正在规划..."，执行阶段显示工具调用，质检阶段显示"正在检查..." | |

---

## T-shirt 初判

**M (5pts)** — 需要重构 agent.py 的核心对话循环，引入多阶段 prompt + 循环控制逻辑。风险在于 prompt 工程复杂度和 token 成本控制。
