# REQ-004: 工作区进程管理

> Intake ID: 2025-05-25-workspace-manager

---

## 原始需求

平台需要管理所有工作区的生命周期：启动、停止、健康检查、端口分配。支持三种启动模式（自动/手动/按需），工作区可隐藏但不直接删除。

---

## 提炼后的需求

### 问题陈述

平台需要一个进程管理器来统一管理所有工作区的生命周期。用户需要灵活控制工作区的启动方式，AI 管家需要能按需唤醒工作区来响应用户对话。

### 范围

**包含（In Scope）：**

- 工作区发现（扫描 workspaces/ 目录）
- 三种启动模式：auto / manual / on-demand
- 进程启停管理（subprocess）
- 端口分配与冲突处理
- 健康检查（HTTP ping）
- 崩溃检测与自动重启（上限 3 次）
- 隐藏/取消隐藏工作区
- 状态 API 暴露给前端

**不包含（Out of Scope）：**

- 工作区创建逻辑（REQ-003）
- 工作区代码生成（REQ-008）
- 硬删除（用户自行删除文件夹）

### 用户故事

**As a** 岛主用户，
**I want** 平台自动管理工作区的启停，并支持按需唤醒，
**so that** 我不需要手动管理进程，常用的自动运行，偶尔用的随时可用。

## 工作区状态模型

### 启动模式（workspace.json 配置）

| 模式 | 字段值 | 含义 | 触发方式 |
|------|--------|------|----------|
| 自动启动 | `"start_mode": "auto"` | 平台启动时自动启动 | 平台进程启动 |
| 手动启动 | `"start_mode": "manual"` | 用户在书架上点击启动 | 用户点击按钮 |
| 按需启动 | `"start_mode": "on-demand"` | AI 对话触发或用户点击时启动 | AI 管家判断需要 / 用户点击卡片 |

### 运行状态（运行时动态）

| 状态 | 含义 | 前端展示 |
|------|------|----------|
| `running` | 正常运行中 | 🟢 绿点 + "运行中" |
| `stopped` | 已停止 | ⚪ 灰点 + "已停止" |
| `starting` | 启动中 | 🟡 旋转 + "启动中" |
| `crashed` | 崩溃（重试耗尽） | 🔴 红点 + "已崩溃" |
| `hidden` | 已隐藏（软删除） | 不显示在书架主列表 |

### 状态流转

```
                    平台启动(auto)
                    用户点击(manual)
                    AI触发/用户点击(on-demand)
                         │
                         ▼
stopped ──────────→ starting ──────────→ running
   ▲                                       │
   │              用户手动停止               │
   └───────────────────────────────────────┘
                                           │
                        进程异常退出         │
                        (重试≤3次)          │
                         │                 │
                         ▼                 │
                      crashed ◄────────────┘
                         │
                    用户手动重启
                         │
                         ▼
                      starting

任意状态 ──用户隐藏──→ hidden
hidden ──用户取消隐藏──→ stopped
```

### workspace.json 扩展字段

```json
{
  "id": "finance",
  "name": "财务助手",
  "icon": "💰",
  "color": "#2563EB",
  "version": "1.0.0",
  "description": "AI 驱动的记账报表",
  "port": 7801,
  "entry": "app.py",
  "python": ".venv",
  "tags": ["财务", "记账"],
  "start_mode": "auto",
  "hidden": false
}
```

## 验收标准

| # | 验收标准 | 可测试性 |
|---|----------|----------|
| AC-1 | 平台启动时，自动扫描 workspaces/ 目录，发现所有含 workspace.json 的子目录 | 放入 3 个工作区目录，启动后 API 返回 3 个 |
| AC-2 | start_mode=auto 的工作区在平台启动后 10 秒内自动启动，健康检查通过 | 检查进程存在 + HTTP 200 |
| AC-3 | start_mode=manual 的工作区平台启动后保持 stopped 状态，用户调用启动 API 后变为 running | 调用 POST /api/workspaces/{id}/start |
| AC-4 | start_mode=on-demand 的工作区在 AI 管家判断需要时自动启动，或用户点击卡片时启动 | 模拟 AI 调用启动接口 |
| AC-5 | 端口冲突时自动跳过已占用端口，分配下一个可用端口（范围 7801-7899） | 占用 7801 后启动工作区，分配到 7802 |
| AC-6 | 工作区进程异常退出后自动重启，最多 3 次，超过后标记为 crashed | kill 进程后观察重启行为 |
| AC-7 | 用户隐藏工作区后，书架 API 不返回该工作区；取消隐藏后恢复显示 | 调用隐藏 API 后验证列表 |
| AC-8 | 平台关闭时，逐个停止所有运行中的工作区，等待 5 秒后强制终止 | 观察平台退出时子进程清理 |
| AC-9 | GET /api/workspaces 返回每个工作区的实时状态（running/stopped/starting/crashed） | 对比 API 返回与实际进程状态 |

## 业务价值

- **用户体验**：用户无需手动管理进程，常用工作区自动就绪
- **资源优化**：on-demand 模式避免低频工作区占用内存
- **稳定性**：崩溃自动恢复，减少用户干预
- **量化**：auto 模式下用户打开浏览器即可使用，零等待
