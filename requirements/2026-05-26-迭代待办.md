1. 保留 git 历史，上游有安全更新/功能迭代时可 `git merge` 拉新

**操作流程：**
```
用户：帮我建一个 XX 工作区
Agent：
  ├─ 搜索 GitHub/Gitee Top 5
  ├─ 展示给用户选择
  └─ 用户确认后：
     1. git clone --depth=1 到 workspaces/xxx/
     2. 添加 workspace.json、schema.sql、db.py
     3. 适配前端为纯 HTML/CSS/JS（无 Node 依赖）
     4. 创建 .venv + 安装依赖
     5. 初始化 data.db
     6. 启动 + 注册到书架
```

## 架构决策：工作区进程管理策略

> 针对"所有项目都单独启动环境和进程"的讨论，给出专业建议。

### 现状
当前每个工作区 = 独立 Python 进程 + 独立 venv + 独立端口（7801-7899）。`workspace_manager.py` 14K 是最大模块之一。

### 问题
- 10 个工作区 = 10 个 Python 进程 + 10 个 venv，内存开销大
- 端口池有限（99 个端口），不适合大规模使用
- 简单工作区（如待办、计数）不值得独立跑一个 uvicorn

### 建议：三层进程模型

```
层级       模式                 适用场景                    venv
─────────────────────────────────────────────────────────────
L1: 轻量   挂载到主进程 ASGI     简单 CRUD、看板、笔记      无独立 venv
           子路由 (/ws/{id})     单表、纯增删改查            共享主环境依赖
─────────────────────────────────────────────────────────────
L2: 标准   独立进程 + 独立端口   大多数工作区                独立 venv
           目前的方式            复杂业务逻辑、自定义依赖    (pyproject.toml)
─────────────────────────────────────────────────────────────
L3: 重型   独立进程 + 独立端口   集成了第三方服务            独立 venv + 额外
           可选择性             Grafana/Jupyter/Streamlit   系统依赖
```

### 具体规则

**何时走 L1（轻量挂载）：**
- schema.sql 不超过 3 张表
- 无外部依赖（只需 fastapi + uvicorn + aiosqlite）
- 前端不超过 3 个页面
- 判断依据：创建工作区时，Agent 评估复杂度，自动选择层级

**何时走 L2（独立进程）：**
- schema.sql 超过 3 张表
- 有额外 Python 依赖（如 pandas、httpx、openpyxl）
- 需要独立端口对外暴露
- 需要独立重启不影响其他工作区

**何时走 L3（重型）：**
- 集成了外部服务（数据库、消息队列等）
- 运行第三方工具（Jupyter、Airflow、Grafana）
- 需要系统级依赖

### 技术实现（L1 方案）

```python
# 主进程挂载轻量工作区
from fastapi import APIRouter

# 轻量工作区不需要独立进程
# 只需在 workspace.json 中声明:
# { "mode": "lightweight", "mount_path": "/ws/{id}" }
# 主进程启动时自动挂载其 app.py 的 router 到主 FastAPI
```

### 优势
- 大量简单工作区零额外开销
- 复杂工作区保持独立隔离
- 用户无需感知层级差异，全自动判断

### Q6 补充（复用主进程）
> 用户：小项目可以判断是否使用主进程，避免无限开销

**实现方式：**
- workspace.json 新增字段 `"mode": "lightweight" | "standard"`
- 创建时 Agent 自动判断模式
- lightweight 工作区注册到主进程的 APIRouter，不占独立端口和进程
- 轻量工作区仍然有独立的数据目录（workspaces/xxx/）和独立的 data.db
