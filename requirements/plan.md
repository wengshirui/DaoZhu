## 新需求详情

### 043 — Gitee 远程控制与聊天

**问题陈述：** 用户目前只能在本地浏览器访问岛主，无法远程操控。需要一种无需公网 IP、无需登录第三方平台的远程交互方式。

**方案：** 利用 Gitee Issues/Webhooks 作为消息通道，实现"用户在 Gitee 发 Issue/评论 → 岛主本地轮询/Webhook 接收 → Agent 处理 → 回复到 Issue"的闭环。

**User Story：**
> As a 岛主用户，I want 通过 Gitee Issue 远程给我的岛发指令并收到回复，so that 我在手机/其他设备上也能操控我的数字小岛。

**验收标准：**
1. 用户在指定 Gitee 仓库发 Issue，岛主在 60s 内识别并处理
2. Agent 处理结果以 Issue 评论形式回复，包含结构化结果
3. 支持基本指令：查看工作区状态、启停工作区、执行简单对话
4. 非法指令（格式错误/无权限）返回友好错误提示
5. 轮询模式下 CPU 占用 < 1%（空闲时）

**T-shirt Size：** M — 跨前后端 + 第三方 API 集成 + 轮询/Webhook 双模式；风险在于 Gitee API 限流和消息时序

**依赖：** Gitee Token（已有）、论坛模块经验（025 已完成）

**灵感来源：** hermes-agent 的 gateway 平台适配器模式（Telegram/Discord/Webhook 等多平台统一接入）

---

### 044 — AutoMovie 方案升级（HTML → Godot）

**问题陈述：** 当前 AutoMovie（话剧导演）是纯 HTML/Canvas 实现，功能受限于浏览器沙箱。升级为 Godot 项目可获得：客户端完全自主、无需浏览器、离线运行、更丰富的 2D/3D 表现力。

**方案：** 将"AI 导演 + 实体演员"系统从 HTML 迁移到 Godot 4.x 项目，AI 部分仍由岛主 Agent 通过 HTTP API 驱动 Godot 客户端。

**User Story：**
> As a 创作者，I want 用自然语言描述剧情后得到一个可独立运行的 Godot 场景文件，so that 我不依赖浏览器、不需要登录任何平台就能创作和分享动画。

**验收标准：**
1. Godot 客户端能接收 JSON 剧本并按序执行实体动作
2. 支持至少 5 种实体类型（人物、门、床、窗、桌）和 10 种动作
3. 导出为独立可执行文件（.exe），双击即可播放
4. AI 导演接口与现有 AGENT.md 中的 JSON 剧本格式兼容
5. 提供"录制"功能，将演出导出为视频文件（mp4）

**T-shirt Size：** L — 新技术栈引入（GDScript）+ 客户端架构设计 + AI↔Godot 通信协议；风险在于 Godot 学习曲线和跨平台导出

**依赖：** Godot 4.x 引擎、现有 AutoMovie 剧本格式

**核心卖点（vs HTML 方案）：**

| 维度 | HTML 方案 | Godot 方案 |
|------|-----------|------------|
| 登录 | 无需 | 无需 |
| 数据归属 | 本地 | 本地 |
| 客户端 | 浏览器 | 独立 exe |
| 离线 | 部分（需 AI API） | 可接 Ollama 完全离线 |
| 表现力 | Canvas 2D | 2D/3D + 粒子 + 音频 |
| 分发 | 发 HTML 文件 | 发 exe 或视频 |

### 045 — 本地 Ollama 模型集成

**问题陈述：** 当前岛主依赖 DeepSeek/OpenAI 云端 API，有网络依赖和隐私顾虑。支持本地 Ollama 后，用户可完全离线使用，数据零外泄。

**方案：** 在配置系统中增加 Ollama provider 选项，Agent 调用时自动路由到本地 Ollama HTTP API（兼容 OpenAI 格式）。

**User Story：**
> As a 注重隐私的用户，I want 配置本地 Ollama 作为 AI 后端，so that 我的所有对话和数据完全不出本机。

**验收标准：**
1. 设置页面可选择"本地模型（Ollama）"作为 AI 后端
2. 自动检测本地 Ollama 服务是否运行，未运行时给出安装引导
3. 列出已安装的模型供用户选择
4. 对话质量在 7B+ 模型下可完成基本工作区操作（启停、查询）
5. 云端 API 不可用时自动降级到本地模型（如已配置）

**T-shirt Size：** M — 需改造 chat_service.py 的 provider 抽象 + 设置页面 UI + 自动检测逻辑；风险在于小模型工具调用能力有限

**依赖：** 用户本地安装 Ollama（可选）

**灵感来源：** hermes-agent 的 model-provider 插件体系（多 provider 统一抽象 + fallback 机制）

## 战略视角：数据主权闭环

这三个新需求共同构成岛主的**数据主权闭环**：

```
┌─────────────────────────────────────────────────┐
│              用户的数字小岛                        │
│                                                   │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐       │
│  │ 本地数据 │   │ 本地模型 │   │ 本地客户端│       │
│  │ (SQLite) │   │ (Ollama) │   │ (Godot)  │       │
│  └────┬────┘   └────┬────┘   └────┬────┘       │
│       │              │              │             │
│       └──────────────┼──────────────┘             │
│                      │                            │
│              ┌───────┴───────┐                    │
│              │   岛主 Agent   │                    │
│              └───────┬───────┘                    │
│                      │                            │
│              ┌───────┴───────┐                    │
│              │  Gitee 通道    │ ← 远程控制入口     │
│              └───────────────┘                    │
└─────────────────────────────────────────────────┘
```

**卖点总结：**
- 不用登录 → Gitee Issue 就是你的遥控器
- 数据是自己的 → SQLite 本地存储，零云端
- 客户端是自己的 → Godot 导出独立 exe
- 有钱的话模型都是自己的 → Ollama 本地推理
